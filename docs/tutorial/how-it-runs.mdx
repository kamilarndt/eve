---
title: "2. Understand the Runtime"
description: "Learn the session, turn, and durable-step model used by the rest of the tutorial."
---

Three terms explain what happened when you sent the first message:

| Term    | Meaning                                                             |
| ------- | ------------------------------------------------------------------- |
| Session | The durable conversation, identified publicly by `sessionId`.       |
| Turn    | One user message and the work it triggers.                          |
| Step    | A checkpoint inside a turn, such as a model call or tool execution. |

eve runs each turn as a durable workflow. We record completed steps so a restart can resume without
repeating finished work. An interrupted step may run again. That distinction matters when a tool
charges a card, sends a message, or performs another external side effect: the tool must be
idempotent even when it also requires approval.

A turn can stop in a waiting state. Approval and question prompts do not hold a server process open. A client later submits the requested input with the current continuation token, and the workflow resumes.

The model loop is framework-owned. You provide instructions and capabilities; eve calls the model,
executes selected tools, records results, and decides whether another model step is required. We
keep this in the runtime so durability and stream events do not depend on application-specific loop
code. One user turn can therefore produce several model calls.

You will use each part of this model in later steps:

- State survives between turns in the same session.
- A subagent runs in a child session.
- Approval parks a turn until you respond.
- Sandbox files survive between turns in the same session.
